# 微服务系列 02 - 架构设计：如何拆分微服务

> 系列文章目录：[微服务从小白到精通](./README.md)

## 1. 微服务拆分的核心原则

微服务的拆分质量直接决定了系统的成败。拆得太细，运维成本爆炸；拆得太粗，等于没拆。

### 1.1 拆分原则一览

| 原则 | 核心思想 |
|------|----------|
| **单一职责** | 每个服务只做一件事 |
| **高内聚低耦合** | 服务内部紧密相关，服务之间松散关联 |
| **限界上下文（Bounded Context）** | 来自 DDD，每个服务有明确的业务边界 |
| **独立部署** | 服务可以独立发布，不影响其他服务 |
| **数据自治** | 每个服务拥有自己的数据存储 |

### 1.2 最关键的原则：限界上下文

限界上下文（Bounded Context）是领域驱动设计（DDD）的核心概念。它定义了一个明确的边界，边界内的模型具有统一的含义。

```
┌──────────────────────────────────────────────────────┐
│                    电商系统                            │
│                                                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │  用户上下文    │  │  订单上下文    │  │  商品上下文    │ │
│  │              │  │              │  │              │ │
│  │ - 用户注册    │  │ - 创建订单    │  │ - 商品管理    │ │
│  │ - 用户认证    │  │ - 订单查询    │  │ - 库存管理    │ │
│  │ - 用户信息    │  │ - 订单状态    │  │ - 价格管理    │ │
│  │              │  │              │  │              │ │
│  │  User 实体   │  │  Order 实体   │  │  Product 实体 │ │
│  │  (用户视角)   │  │  (订单视角)   │  │  (商品视角)   │ │
│  └──────────────┘  └──────────────┘  └──────────────┘ │
└──────────────────────────────────────────────────────┘
```

注意：同一个"商品"概念在不同上下文中含义不同：
- 商品上下文：Product = 名称、描述、规格、价格
- 订单上下文：Product = 订单行项目、购买数量、成交价格
- 库存上下文：Product = SKU、库存数量、仓库位置

## 2. 拆分方法

### 2.1 按业务能力拆分（推荐）

按业务能力拆分是最主流的方法，来源于 DDD 的限界上下文概念。

**步骤：**
1. 识别业务领域和子域
2. 找出核心域、支撑域、通用域
3. 为每个子域定义限界上下文
4. 每个限界上下文对应一个微服务

**电商系统示例：**

```
核心域（直接产生业务价值的部分）：
├── 商品域 → product-service
├── 订单域 → order-service
└── 支付域 → payment-service

支撑域（支撑核心业务的部分）：
├── 用户域 → user-service
├── 物流域 → logistics-service
└── 通知域 → notification-service

通用域（所有业务都需要的部分）：
├── 认证授权 → auth-service
└── 文件存储 → file-service
```

### 2.2 按子域拆分

子域（Subdomain）是 DDD 中对业务领域的进一步细分：

| 子域类型 | 说明 | 示例 |
|----------|------|------|
| **核心子域** | 业务核心竞争力，最复杂的部分 | 订单流程、推荐算法 |
| **支撑子域** | 不是核心但必不可少 | 用户管理、消息通知 |
| **通用子域** | 通用解决方案，可外购 | 认证授权、邮件发送 |

### 2.3 按数据拆分

识别数据的所有权，每个服务拥有自己的数据：

```
错误做法：
┌──────────┐     ┌──────────┐
│ 订单服务  │────▶│ 共享DB    │◀────│ 用户服务  │
└──────────┘     └──────────┘     └──────────┘

正确做法：
┌──────────┐     ┌──────────┐     ┌──────────┐
│ 订单服务  │     │ 用户服务  │     │ 商品服务  │
│ ┌──────┐ │     │ ┌──────┐ │     │ ┌──────┐ │
│ │Order │ │     │ │User  │ │     │ │Product│ │
│ │  DB  │ │     │ │  DB  │ │     │ │  DB  │ │
│ └──────┘ │     │ └──────┘ │     │ └──────┘ │
└──────────┘     └──────────┘     └──────────┘
```

## 3. 实战：电商系统拆分

### 3.1 事件风暴（Event Storming）

事件风暴是一种快速梳理业务流程的方法：

```
橙色卡片 = 领域事件（过去时态）
蓝色卡片 = 命令（触发事件的动作）
黄色卡片 = 聚合（处理命令的实体）
紫色卡片 = 外部系统
粉色卡片 = 策略/规则

排列示例：

用户下单 → [OrderCreated] → 库存扣减 → [StockDeducted] → 支付处理 → [PaymentCompleted]
     ↑                                                              ↓
  [Command]                                                    发货
  创建订单                                                 [OrderShipped]
```

### 3.2 电商系统最终拆分

基于事件风暴，电商系统拆分为以下服务：

## 4. 服务间通信方式选择

### 4.1 同步通信 vs 异步通信

| 维度 | 同步通信 | 异步通信 |
|------|----------|----------|
| **方式** | REST、gRPC、Feign | 消息队列（RabbitMQ、Kafka） |
| **优点** | 简单直接、实时响应 | 解耦、削峰填谷、可靠 |
| **缺点** | 级联故障、等待延迟 | 复杂、最终一致性 |
| **场景** | 查询类请求 | 命令类请求、事件通知 |

### 4.2 选择指南

```
需要实时响应？
├── 是 → 用同步（REST/gRPC）
│        ├── 服务间简单调用 → REST + Feign
│        └── 高性能场景 → gRPC
└── 否 → 用异步（消息队列）
         ├── 简单消息 → RabbitMQ
         ├── 大数据流 → Kafka
         └── 事件溯源 → Kafka + Event Store
```

## 5. 数据库拆分策略

### 5.1 拆分模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **每个服务一个数据库** | 每个服务独享自己的数据库 | 标准微服务 |
| **每个服务一个 Schema** | 同一数据库实例，不同 Schema | 过渡阶段 |
| **共享数据库** | 多服务共享数据库 | 反模式，应避免 |

### 5.2 数据一致性方案

跨服务数据一致性是微服务最大的挑战之一：

| 方案 | 一致性 | 复杂度 | 适用场景 |
|------|--------|--------|----------|
| **2PC** | 强一致性 | 高 | 传统分布式事务 |
| **TCC** | 最终一致性 | 高 | 资金类业务 |
| **Saga** | 最终一致性 | 中 | 长流程业务 |
| **事件驱动** | 最终一致性 | 中 | 通知类场景 |

> 详细的分布式事务方案将在 [09 - 分布式事务](./09-distributed-transaction.md) 中深入讲解。

## 6. 常见反模式（Anti-patterns）

### ❌ 反模式 1：纳米服务

服务拆得太细，一个 CRUD 操作就是一个服务，导致：
- 服务数量爆炸
- 运维成本极高
- 网络开销大

### ❌ 反模式 2：共享数据库

多个服务直接访问同一个数据库：
- 违反数据自治原则
- 服务间隐性耦合
- 数据库变更影响多个服务

### ❌ 反模式 3：分布式单体

服务虽然拆分了，但调用链路复杂，部署仍然需要协调：
- 改一个功能需要同时改多个服务
- 任何服务挂掉整条链路不通
- 本质上还是单体，只是通过网络调用

### ❌ 反模式 4：API 网关做业务逻辑

网关承担了过多业务逻辑（数据聚合、转换等）：
- 网关变成性能瓶颈
- 违反网关的定位（路由、鉴权、限流）

## 7. 拆分的时机与节奏

### 7.1 渐进式拆分

不要一次性把所有模块都拆成微服务，建议：

```
第一阶段：单体应用
    ↓ （业务验证完成，需要团队扩展）
第二阶段：提取最独立的模块（如通知服务）
    ↓ （遇到部署瓶颈）
第三阶段：提取核心服务（用户服务、商品服务）
    ↓ （业务增长，需要独立扩展）
第四阶段：全面微服务化
```

### 7.2 绞杀者模式（Strangler Fig）

像藤蔓绞杀大树一样，逐步用新服务替代旧系统：

```
阶段 1：新功能用微服务实现，旧功能保持不变
阶段 2：逐步将旧功能迁移到微服务
阶段 3：旧系统完全被替代

┌──────────────┐     ┌──────────────┐
│   旧系统      │     │   新服务 A    │
│  (逐步缩减)   │     │  (逐步增长)   │
└──────────────┘     └──────────────┘
         ↑                   ↑
         │    API Gateway     │
         └───────┬───────────┘
                 │
             Client 请求
```
