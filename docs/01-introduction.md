# 微服务系列 01 - 微服务概述：从单体到微服务的演进

> 系列文章目录：[微服务从小白到精通](./README.md)

## 1. 什么是微服务？

微服务（Microservices）是一种架构风格，它将一个大型应用程序拆分为一组小型、独立的服务，每个服务：

- **围绕业务能力构建**：每个服务专注于一个特定的业务功能
- **独立部署**：每个服务可以独立地构建、测试和部署
- **松耦合**：服务之间通过明确定义的 API（REST、gRPC、消息队列）进行通信
- **自治**：每个服务拥有自己的数据库、业务逻辑和技术栈

### 1.1 微服务的核心特征

```
┌─────────────────────────────────────────────────────┐
│                    微服务架构                         │
├─────────────────────────────────────────────────────┤
│                                                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
│  │ 用户服务  │  │ 订单服务  │  │ 商品服务  │           │
│  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │           │
│  │ │  DB  │ │  │ │  DB  │ │  │ │  DB  │ │           │
│  │ └──────┘ │  │ └──────┘ │  │ └──────┘ │           │
│  └──────────┘  └──────────┘  └──────────┘           │
│       ↕             ↕             ↕                   │
│  ┌──────────────────────────────────────────┐        │
│  │              API Gateway                  │        │
│  └──────────────────────────────────────────┘        │
│                       ↕                               │
│                  Client / App                         │
│                                                       │
└─────────────────────────────────────────────────────┘
```

Martin Fowler 和 James Lewis 在 2014 年的文章中总结了微服务的九大特征：

| 特征 | 说明 |
|------|------|
| **组件化与服务** | 通过服务而非库来实现组件化 |
| **围绕业务能力组织** | 团队按业务领域划分（康威定律） |
| **产品而非项目** | 团队对产品的整个生命周期负责 |
| **智能端点与哑管道** | 服务间通信使用简单的 REST/消息，不加复杂的智能路由 |
| **去中心化治理** | 每个团队可以选择适合自己的技术栈 |
| **去中心化数据管理** | 每个服务管理自己的数据库 |
| **基础设施自动化** | CI/CD、容器化、自动化部署 |
| **容错设计** | 假设任何服务都可能失败，设计时考虑容错 |
| **演进式设计** | 架构随业务演进而非一步到位 |

## 2. 从单体到微服务的演进

### 2.1 单体架构（Monolith）

单体架构是最传统的应用架构方式，所有功能都打包在一个可部署单元中。

```
┌────────────────────────────────────────┐
│              单体应用                    │
│  ┌────────┬────────┬────────┬────────┐ │
│  │ 用户   │ 订单   │ 商品   │ 支付   │ │
│  │ 模块   │ 模块   │ 模块   │ 模块   │ │
│  └────────┴────────┴────────┴────────┘ │
│  ┌────────────────────────────────────┐ │
│  │          共享数据库                  │ │
│  └────────────────────────────────────┘ │
└────────────────────────────────────────┘
```

**单体架构的优点：**
- 开发简单：IDE 中一个项目搞定
- 部署简单：打一个包部署即可
- 测试简单：端到端测试方便
- 事务管理简单：本地事务即可

**单体架构的痛点（随着规模增长）：**
- 🐌 **代码膨胀**：百万行代码，编译慢、启动慢
- 🔒 **耦合严重**：改一个小功能可能影响整个系统
- 🚀 **部署风险高**：一次部署涉及所有功能
- 👥 **团队协作难**：多人修改同一代码库，冲突多
- ⚡ **扩展困难**：无法对单个模块独立扩容
- 🔧 **技术栈僵化**：难以引入新技术

### 2.2 演进过程

架构不是一蹴而就的，通常经历以下阶段：

```
单体架构 → 模块化单体 → SOA → 微服务 → 云原生微服务
```

| 阶段 | 特点 | 适用场景 |
|------|------|----------|
| **单体架构** | 所有功能一个应用 | 初创期、MVP |
| **模块化单体** | 代码模块化，仍单部署 | 中小型团队 |
| **SOA** | 服务化，ESB 集成 | 企业级集成 |
| **微服务** | 独立部署，去中心化 | 大型互联网应用 |
| **云原生微服务** | 容器化+K8s+Service Mesh | 超大规模系统 |

> ⚠️ **重要提醒**：不是所有项目都适合微服务！微服务引入了分布式系统的复杂性，只有当系统复杂度达到一定程度时，微服务才能带来净收益。

## 3. 微服务的优势与挑战

### 3.1 优势

| 优势 | 详细说明 |
|------|----------|
| **独立部署** | 每个服务可以独立发布，不影响其他服务 |
| **技术异构** | 不同服务可以使用不同的编程语言、数据库 |
| **弹性扩展** | 可以只对高负载的服务进行水平扩展 |
| **故障隔离** | 一个服务挂掉不会导致整个系统崩溃 |
| **团队自治** | 小团队独立负责一个或几个服务，效率更高 |
| **渐进演进** | 可以逐步替换旧服务，不影响整体系统 |

### 3.2 挑战

| 挑战 | 详细说明 |
|------|----------|
| **分布式复杂性** | 网络延迟、数据一致性、分布式事务 |
| **服务发现** | 服务实例动态变化，如何找到对方？ |
| **数据一致性** | 跨服务的数据一致性如何保证？ |
| **运维复杂度** | 大量服务需要监控、日志、追踪 |
| **测试困难** | 集成测试、契约测试更复杂 |
| **团队沟通** | 服务边界划分需要跨团队协商 |

## 4. 什么时候该用微服务？

### 4.1 适合微服务的场景

✅ 系统已经非常复杂，单体难以维护  
✅ 团队规模较大（10+ 开发者）  
✅ 不同模块有不同的扩展需求  
✅ 需要技术栈的灵活性  
✅ 已有成熟的 DevOps 和自动化部署能力  
✅ 业务领域边界清晰

### 4.2 不适合微服务的场景

❌ 初创 MVP 阶段，快速验证为主  
❌ 团队很小（3-5 人）  
❌ 业务逻辑简单，模块间紧密耦合  
❌ 缺乏 DevOps 基础设施  
❌ 没有领域驱动设计经验

## 5. 主流微服务技术栈

### 5.1 按语言/框架分类

| 技术栈 | 语言 | 核心组件 | 适用场景 |
|--------|------|----------|----------|
| **Spring Cloud** | Java | Eureka/Gateway/Config/Sleuth | 企业级 Java 应用 |
| **Spring Cloud Alibaba** | Java | Nacos/Sentinel/Seata | 国内 Java 生态 |
| **Go Micro** | Go | Go-Micro/Consul | 高性能后端 |
| **Dubbo** | Java | Dubbo/Zookeeper | 高性能 RPC |
| **Kratos** | Go | Kratos/Consul | Go 微服务 |
| **.NET** | C# | Steeltoe/Ocelot | .NET 生态 |
| **Node.js** | TypeScript | NestJS/Seneca | 前端团队扩展后端 |

### 5.2 本系列选择的技术栈

本系列以 **Spring Boot 3.x + Spring Cloud 2023.x** 为主，辅助介绍 **Spring Cloud Alibaba** 和其他替代方案：

```
Spring Boot 3.2+         →  基础框架
Spring Cloud 2023.x      →  微服务全家桶
  ├── Eureka             →  服务注册与发现
  ├── Spring Cloud Gateway → API 网关
  ├── Spring Cloud Config →  配置中心
  ├── OpenFeign          →  声明式服务调用
  ├── Resilience4j       →  熔断与容错
  └── Sleuth + Zipkin    →  分布式追踪

Spring Cloud Alibaba
  ├── Nacos              →  注册中心 + 配置中心
  ├── Sentinel           →  流量控制与熔断
  └── Seata              →  分布式事务

Docker + Kubernetes      →  容器化与编排
```

## 7. 关键概念速查

| 概念 | 一句话解释 |
|------|-----------|
| **服务注册** | 服务启动时把自己的地址告诉注册中心 |
| **服务发现** | 调用方从注册中心获取目标服务的地址 |
| **API 网关** | 统一入口，负责路由、鉴权、限流 |
| **熔断器** | 当下游服务异常时快速失败，防止级联故障 |
| **链路追踪** | 跟踪一个请求在多个服务间的调用链路 |
| **配置中心** | 集中管理所有服务的配置，支持动态刷新 |
| **服务网格** | 用 Sidecar 代理处理服务间通信，应用无感知 |
