# 微服务系列 22 - 生产就绪检查清单

> 系列文章目录：[微服务从小白到精通](./README.md)

## 1. 为什么需要生产就绪检查？

一个微服务能跑起来不等于可以上生产。生产环境需要考虑可靠性、可观测性、安全性、性能等方方面面。本篇提供一个系统性的检查清单，确保你的微服务"生产就绪"。

```
开发环境：功能能跑就行
测试环境：功能验证通过
预发环境：和生产一致，验证非功能性需求
生产环境：7×24 小时稳定运行 ← 这需要额外的工作量
```

## 3. 可靠性检查

### 3.1 容错

| # | 检查项 | 说明 | 状态 |
|---|--------|------|------|
| 1 | 所有外部调用有超时 | Feign/RestTemplate 设超时 | ☐ |
| 2 | 所有外部调用有重试 | 最多 3 次，带指数退避 | ☐ |
| 3 | 所有外部调用有熔断 | Resilience4j/Sentinel | ☐ |
| 4 | 熔断有降级策略 | 返回兜底数据而非报错 | ☐ |
| 5 | 关键调用有限流 | 防止突发流量压垮系统 | ☐ |
| 6 | 写操作幂等 | 重复请求结果一致 | ☐ |
| 7 | 异步消息可重试 | 消费失败进入重试队列 | ☐ |
| 8 | 死信队列处理 | 重试失败进入 DLQ + 告警 | ☐ |

### 3.2 优雅上下线

| # | 检查项 | 说明 | 状态 |
|---|--------|------|------|
| 9 | 优雅停机 | `server.shutdown=graceful` | ☐ |
| 10 | K8s preStop 钩子 | sleep 等待流量排空 | ☐ |
| 11 | 健康检查三件套 | Startup + Liveness + Readiness | ☐ |
| 12 | 启动探针给足时间 | 慢启动应用需长 initialDelay | ☐ |
| 13 | 注册中心主动下线 | 停机前从 Eureka/Nacos 注销 | ☐ |

### 3.3 数据

| # | 检查项 | 说明 | 状态 |
|---|--------|------|------|
| 14 | 数据库连接池配置 | 合理的 min/max 连接数 | ☐ |
| 15 | 数据库重连机制 | 连接断开后自动重连 | ☐ |
| 16 | 事务边界明确 | @Transactional 范围最小化 | ☐ |
| 17 | 数据备份策略 | 定时备份 + 恢复验证 | ☐ |

## 9. 自评打分

对每个检查项打分（0-2），总分 140：

```
分数 = Σ(每项得分)

0   = 未考虑
1   = 部分实现
2   = 完整实现

评级：
0  - 50  🔴 风险极高，不建议上生产
51 - 90  🟡 需要改进，存在风险
91 - 120 🟢 基本就绪，小项可以跟进
121- 140 ✅ 生产就绪
```

## 10. 渐进式落地

不要试图一次搞定所有检查项，按优先级逐步落地：

```
第一优先级（上线前必须）：
├── 超时和重试
├── 健康检查
├── 结构化日志 + Trace ID
├── HTTPS + 认证
└── 告警

第二优先级（上线后一周内）：
├── 熔断降级
├── 限流
├── 指标监控（Prometheus + Grafana）
├── 优雅停机
└── 幂等设计

第三优先级（持续改进）：
├── 分布式追踪
├── 混沌测试
├── 容器安全扫描
├── 性能基线
└── 容量规划
```
